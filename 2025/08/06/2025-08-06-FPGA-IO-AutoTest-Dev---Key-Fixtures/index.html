FPGA IO Auto Test Dev - Key Fixtures

Brief introduction of test process

start - run pytest - @pytest.mark.IO - run test function - parameters correspond to the py_conftest operation fixtures (call fixture/method - execute function) - perform the actual test actions (such as reset, download, etc) - each fixture completed - end

For example:

Key Fixtures

Logger

The logger fixture is the dedicated log manager for each test case. It ensures that logs and reports are accurate, clean, and automated, thereby enhancing the speed of troubleshooting and the efficiency of team collaboration.

1.Automate the log initialization for each use case/method

If the scope of ‘fixture’ is set to ‘function’, it will be automatically called before each test case is run.

2.The file management of logs and reports is isolated from the output

The log files and sheets for each use case are unique. Different test cases’ logs will not be confused.

3.Automatic standardization of the log process

4.Automatic judgment and conclusion after the end of the test cases

5.Facilitates integration and enhancement

You can directly use this logger to record logs, write test steps, update Excel, etc. in any use case. There is no need to repeatedly write the initialization and closing code.

DUTx

It is used to manage the initialization and cleanup of the resources of the test target device (Device Under Test) named DUTx. Its core responsibility is to enable each test case to safely, repeatedly and independently obtain a “pre-prepared DUTx device” for the current test, and properly release the resources after the test case is completed. The DUTx fixture provides “highly integrated device instances”.

Use case begins: Prepare a complete DUT1 specifically for this use case;
After the use case is completed: Automatically perform the post-processing cleanup of the equipment communication channel.

It arrives as an object of the SocDevice type (or a similar device entity).

The “get_device_info” function is capable of obtaining and encapsulating various parameter information related to devices, such as IC, part_number, fpga_proj, and so on.

get_device_info - Method for obtaining parameters

You have passed a large number of parameters to the get_device_info function (such as IC, part_number).
Inside the fixture, by using __get_device_info(role), based on the device role (such as “DUTx”), combined with the global config, test case config, role definition, and function parameters, the parameter combinations for each device are determined.

The main parameters are derived from three categories:
Global configuration file (such as the yaml/toml/python configuration pointed to by dev_config_file) ;
The parameters of the pytest fixture (such as test type, test configuration, log path, etc. are all provided by the external fixture) ;
Example-level/role-level specific configuration (parameters such as IC and part_number under the role can be overridden in tc_test_config.get_test_part()).

do_image_download

1.First, call the function check_fpga_status to check the current power status of FPGA platform. Before downloading, it is necessary to ensure that the FPGA has been powered on.
If the power is not turned on (returns False): Print “FPGA power off”, and then attempt to power on (call do_fpga_power_on()).

2.Automatically modify and update the configuration file (user_settings.config) of the burning tool, and update the path of Keil (or other tools) used for burning required by the current process.

Check whether the file “auto_download_bat” exists (commonly seen in automatic burning scripts / batch processing scripts). If it doesn’t exist, log the information and return False, terminating the process.

3.Build the cmd to download image

This is the sequence of commands that will be passed to the operating system and executed by the CMD window. is the name of the FPGA engineering or image file that needs to be burned.

The purpose is to use the script to automatically download/program the FPGA image.

4.try: xxx except xxx
Responsible for automatically invoking the burn-in script (i.e., the cmd script constructed above), asynchronously collecting its output and execution results, and determining whether the burn-in was successful based on the return value and queue results - any error in any of these steps will be detailedly recorded in the log, facilitating later problem location.

5.finally：

Regardless of whether the script burning process is successful or encounters an error, the finally block will always be executed to ensure that log collection does not miss any data.
This step automatically “backups and organizes the log files generated by the script output”, ensuring that the detailed logs of each burn operation are properly saved in separate directories with timestamps, making it convenient for subsequent location, comparison, and problem tracing.
