How to add test case for ICx

Prerequisites

Tool:

Integration Builder, an Integrated tool to build images for devices under test.

Integration Tester, an Integrated tool for burning images and running automated test cases, not open source;

Python 3.10, for coding.

Add test case for an existed module_A

There are two types to add test case: “by test case” and “by module”.

By test case

(1) Integration Builder

Integration Builder
│  ├─xxx
│  ├─config
│  │  ├─module_A
│      │  └─CASEABC/Test_CASEABC.py
│  ├─build_sequence
│  │  ├─-.xml
│  ├─xxx

Take CASEABC test case in for example, we need to add the following two key files in tool:

1.config//CASEABC/Test_CASEABC.py

The configurations for tool to do build.

Based on the different cores of ICx, create different classes, for example, class Test_CASEABC___, and configure the corresponding information, such as repo_path, target core, bin_dir, …. , for specific role. It also supports modifying related macros.

2.build_sequence/-.xml

Configure the required running cases through XML.

      Test_CASEABC     ---

(2) Integration Tester

Integration Tester
│  ├─xxx 
│  ├─config
│  │  ├─module_A
│      │  └─CASEABC/Test_CASEABC.py
│  ├─test_module
│  │  ├─module_A
│      │  └─CASEABC/Test_CASEABC.py
│  ├─test_sequence
│  │  ├─-.xml
│  ├─xxx

Take CASEABC test case in for example, we need to add the following tree key files in tool:

1.config//CASEABC/Test_CASEABC.py

The test configuration of the test case.

Based on the different cores of ICx, create different classes, for example, class Test_CASEABC___, and configure the corresponding information, such as IC name, project name, LOG_Check if needed, LOG_Check_timeout if needed, etc.

2.test_module//CASEABC/Test_CASEABC.py

The test script for CASEABC case of Module_A.

Create class Test_CASEABC(TestByCase) to achieve the entire testing process, the class name must be the same as the name of the Py module - Test_CASEABC.py.

The functions of Test_CASEABC class:

1> Get test case description and step definition by case_description() method;

2> Dynamically obtain test configuration by get_test_config(get_configure_parameters) method;

3> Complete the testing actions and verification process by do_test() method;

4> Testing interfaceization: test(…) The method is the actual interface that pytest actually calls.

pytest find Test_CASEABC.test() case
↓
pytest automatically calls test(), and fixture parameters are injected.
↓
The test() function calls do_test(…) to implement a set of actual test actions and verifications.
↓
step 1: Log capture and configuration
step 2: DUTx reboot & downoad action
step 3: Log waiting and comparison
↓
Back to test(), and finish the process
↓
assert self.result  → Decide on the pass/fail outcome of the test case.

Summary

Pytest automatically searches for all test cases based on the test_xx.py file and the test_xx functions/classes. Class Test_CASEABC falls under the typical category of “Class Use Case Entity”, inherited from TestByCase, it indicates that it has acquired common testing auxiliary functions (such as loop mechanism, result recording, interface packaging, etc.), thereby enhancing code reusability and standardized management.

3.test_sequence/-.xml

Configure the required running cases through XML.
