Fake Copy Mechanism for Protecting Project

Created2025-10-19|Updated2025-10-19
|Post Views:

Introduction

“Fake Copy” (pseudo-copy) typically refers to a situation in software or system operations where a “copy” action is simulated or emulated, but in reality, the data/file is not physically copied to the target location. This kind of operation usually occurs in scenarios such as automated testing, virtual file systems, or engineering scripts.

This blog mainly introduces the “Fake Copy mechanism for protecting project” implemented in the automated testing Groovy script. It is a sophisticated design used to handle code frozen images, aiming to solve the version consistency issues in parallel development.

What is a “protected item”

In large embedded projects (such as this XX SDK), some core modules are read-only or protected, for example:
Chip underlying driver
Bluetooth protocol stack
Operating system kernel patch

These modules are maintained by specific team and should not be modified by ordinary application developers. In the CI process, these projects are defined as “protected projects”.

The problems solved by the Fake Copy mechanism

The test process is as follows:
1.Initial test build: Build the test firmware based on the latest code and freeze this version (save as an image).
2.Test execution: Conduct the test using the frozen firmware (this may take a very long time).
3.Bug Fixing: The developers continued to submit code to fix bugs during testing phase.
4.Re-test: It is necessary to verify whether the fix is effective.

Question: If the latest code is directly used during the re-test, then:
The protected project may have been updated, resulting in changes to the test environment.
It is hard to determine whether the test failure was caused by the fix or by changes in other modules.

4. Example of actual work process

Suppose we have the following projects:
lowerstack-patch (Protection Project) - Bluetooth Protocol Stack
sdk (protection project) - chip SDK
my-app (non-protected project) - Application

Scene: After fixing the bugs in my-app, it is necessary to conduct a retest.

A.Initial state

B.Carry out Fake Copy

// Roll back the protected project to the frozen version
cp -r /lowerstack-patch /lowerstack-patch // commit_D → commit_A
cp -r /sdk /sdk // commit_E → commit_B
// The non-protected project “my-app” remains at the latest version commit_F

C.Construction result

The test firmware is based on: lowerstack-patch@commit_A + sdk@commit_B + my-app@commit_F
Ensure that only the changes in my-app are tested, thereby eliminating the interference from changes in the underlying modules.

5. Why is it called “Fake” Copy?

It is not a genuine Git operation (such as checkout, reset).
“Simulate” a specific code version state through file system-level replication.
Transparent to the Git repository, without altering the actual Git history or references.

This mechanism is highly practical in the continuous integration of large-scale embedded systems, effectively balancing development efficiency and test reliability.

Values

Version consistency: Ensuring the stability and reliability of the testing environment.
Problem Isolation: Accurately Identify the Source of the Problem (Application Layer vs. Underlying Layer).
Parallel development: Enables the lower-level and application-level teams to work concurrently without interference.
Testing reliability: Prevent fluctuations in test results due to environmental changes
