Jenkins-CI-CD groovy file summary

Created2025-10-12|Updated2025-10-12
|Post Views:

Introduction

Groovy file, Jenkins Pipeline script, used to define the processes of automated building, testing, deployment and report.

Work Flow and Main Functions

Initialization -> build (do_ci_build) -> test build (do_test_build) -> test execution (do_ci_test) -> issue

Initialization

Define key variables and constants, such as

build (do_ci_build)

1.Main objective: Basic compilation verification
2.Scope of construction: Perform basic CI compilation and verify the compilability of the code.
3.Build command: python script/build/ci_build.py –chipType ${chip_type}
4.Build type: Basic image build
5.Test correlation: Does not involve the generation of test binary files
6.Environment requirements: relatively simple, mainly for verifying the compilation environment

test build (do_test_build)

Main work:
Obtain version info of sub-module
Set the name of test script
Copy test files and scripts
Build test image
Prepare test environment

1.Main objective: To prepare a complete binary file for testing
2.Build scope: Complete test image build, including:
- Special handling of the protected project
- Various construction types:

3.Build command: A more complex build command that includes modular test slicing
4.Special handling:
- Handling of code freeze images
- The fake copy mechanism of the protection project
- Branch mapping processing

5.Test Preparation:
- Copy the trace file
- Prepare the test script
- Generate the binary files required for the test

Difference between do_ci_build and do_test_build

The CI build focuses on “whether it can be compiled”, while the test build focuses on “whether it can be tested”. The latter requires the preparation of a complete testing ecosystem. And do_test_build prepares “firmware for testing”.

test execution (do_ci_test)

Parallel test: Multiple testing modules run simultaneously
Environmental inspection: Automatic diagnostic test for environmental issues
Result collection: Generate test reports and logs
Allure Report: Generating Beautiful Test Reports

issue tracking

Jira Integration: Automatically Create Issue Tickets
Result record: Update test database
Notification mechanism: Feedback results through Gerrit comments

Key function modules

manage test modules

Intelligent testing decision-making

Based on the submitted commit info, determine whether testing is necessary:
Skip the tests for the changes related to spec, debug and doc, etc..
Only specific modules (such as bt_audio_trx and danshboard) need to be tested.

Environmental fault-tolerance mechanism

Automatically retrying failed tests related to the environment, but not retrying logic errors excessively.

Multi-IC support

Key features

Parallel testing architecture

Support parallel execution of multiple test cases to enhance efficiency.

Version Control Integration

Precisely record the submission version of each sub-module.

Intelligent Resource Management

Control the use of resources through throttling.

Complete Reporting System

Allure Report: Visualized Test Results
Jira Integration: Automatically Create Defect Tickets
HTML Report: Release Test Summary
Gerrit comment: Real-time feedback results

Applicable scenarios

Code submission verification: Triggered by Gerrit code submission
Continuous Integration: Ensuring Code Quality
Multi-platform testing: Supports different chip configurations
Automated testing: Reducing the workload of manual testing

Summary

This assembly line embodies the best practices of modern CI/CD, providing reliable automated testing support for the development of IC.
