FPGA Platform AutoTest Tool Dev

Created2025-09-26|Updated2025-09-26

Introduction

Entire test process

Update SDK source code -> Download and compare bitfile -> Compile -> Burn bitfile and image -> Execute the test -> Record test result -> Upload test log -> Report Jira

Efforts

1.Design and customize auto test tool for Platform module;
2.Write local py script to connect the whole process above;
3.Modify and call integration tools to finish compilation and test, need to integrate FPGA_Platform module and new test cases;
4.Tool delivery, onsite support and debug;
5.Tool update and maintaining.

Build by case

key variables, methods or classes

1.xml file: Specify test cases to be build;
2.need_build_item: Specify the project to be compiled, like ROM and lowerstack, etc.
3.Test__>_partNumber class:

  • Find the corresponding class by parsing the “TC name” in the XML;
  • Define class method for each project to be compiled;
  • Do configurations for each project, like project path, macro modification, add annotation, and generations’ path and prefix, etc.

build steps

Configure -> Build -> Generations Collection

Path preparation

Set the main folder of module output: The base path for storing all generations after building.

Traverse the TCs and build

1.Obtain tc info, like tc_name.
2.Print the separator for current tc and mark the building start.
3.Configure initialization:

  • Obtain the build_config_class, and config_type (ByTc or ByModule);
    API: get_tc_build_configuraion
  • Initialization build_config_class instance;
  • Load ic specified info;
  • Set ic name;
  • Set submodule path;
  • Determine whether there are any differences in roles in current configuration.
    4.Output Dir: get_build_output_dir.
    5.Obtain all components (in list form), which are necessary for this build.
    API: get_soc_mandatory_build_part
    Traverse the items list (name, method, path), and for each item:
  • Print a separator;
  • Call the corresponding build method, such as build_item_func, like build_np_rom();
  • Pass in the relevant parameters;
  • If the build fails, skip it.

Generations collection

1.Prepare default bin and trace files.
API: copy_default_bin_and_trace
2.Print all building generations for confirmation.

Test by case

Please see How-to-add-new-test-case-for-ICx blog for more pytest process. In this blog, I’ll take Test_CASE.py for example.

Entry function and parameter handling

Call do_test() in main function:
1.Use argparse to parse command-line parameters.
2.Environmental Preparation: Parameter verification, create necessary directories, etc.
3.Initialize test process object and instantiate the test process class.
4.Start pytest test

Pytest loading and tc discovery

After pytest is launched, it will automatically scan the specified test_module directory.
It will find all test_*.py test cases (such as test_CASE.py).
It will load the test case classes (such as Test_CASE_XX_XX), and configure environment according to the command line parameters.

Test case execution process

Initialization

The use case class inherits from TestByCase
External configuration is passed through command line parameters, etc..

Test step

The do_test_steps method: a complete test loop

  1. Obtain configuration (LOG_Check, timeout time, etc.).
  2. Complete core operations such as power-on, status check, image download, and reset for the FPGA DUT device.
  3. Set up log capture and inspection, and determine if the specified keywords appear in sequence.
  4. Execute specific action do_reset_after_download.
  5. Verify the log results and determine if the test is successful.

test method: The actual entry point for Pytest to run

  1. logger sets the execution stage.
  2. Call the above do_test_steps method.
  3. Summarize the results (self.finish), and assert the results.

How parameters are passed during test process

When pytest executes test cases, based on fixtures or parsing parameters, it will automatically pass the parameters in the command line (such as –IC, –logDir, –tcList, etc.) to the test method and related dependencies in test_CASE.py.
Test results, logs, reports, etc. are generated in the directory specified by the parameters(such as outputDir).

Return test result and exit the main program

The do_test() function returns test result.
The main entry point checks the return value and uses sys.exit(0/1), which is suitable for integrating into CI/continuous integration pipelines.

Define the CheckLogContainsInOrderParams class and declare “I want to check these logs in order.”
Call the test interface CheckLogContainsInOrderParams and pass in parameters.
Register the log monitor (such as watcher/observer, etc.).
After log collection, trigger the log checking logic.
Traverse the actual matching code in the log and compare it with key_words in order.

2.How to unify the project compilation structure for different modules of the same chip?
During compilation process, in the underlying get_soc_mandatory_build_part function, different modules define different need_build_item, but ensure that same build_item uses same method/API (like build_rom, build_lowerstack, etc.). In the outermost specific Test_CASE.py, with the same interface name (like get_rom_diff, get_lowerstack_diff, etc.), make different configurations, such as different project paths.
