ICx FPGA IO AutoTest Dev

Goal

Connect all the automated tests for a certain IC at the FPGA stage together by using local automation scripts.

Process

Test Type: LOCAL

Tool: Python

Note: Make full use of the evening time for this process. No human intervention is required throughout.

1. West Project

Make sure you have installed west and cloned source codes already.

Clean all the git directories under west project and run ‘west update’.

2. Print commit info and save to local yml file

Print the commit info for each package under the west project of this test, and save to local yml file.

3. Clear bit file

Clear the local storage path where historical bit files were downloaded.

4. Download bit file

Download the latest bit file to local from Wiki.

5. Generate history.log

Generate the history.log for this test, including the date, bit file version and west info.

6. Call Integration Tools

Call the integration tools recursively for the ap and np dual-core:

1> Call Integration Builder

​ Call IntegrationBuilder tool to build image

2> Call Integration Tester

​ Call IntegrationTester tool to burn bit file and image sequentially to FPGA platform.

3> Run test case

​ The corresponding test cases of each sub-module are executed in a loop (RD can flexibly configure the sub-modules to be tested for different cores).

7. Upload test log to FTP

Upload the test logs to the FTP server after all the tests are completed.

8. Report Jira

Push Jira bugs separately for ap and np, and assign them to the corresponding RD based on different sub-modules.

Implementation

1. West Project

Call the existing API to complete the “clean” function, and then run “west update” command to complete updating operation.

2. Print commit info and save to local yml file

1> Write print_commit_info_and_save function to achieve step 2:

Call get_west_project_info and set_west_project_info functions to get and write commit info to *.txt first;

Iterate through the list and print the commit info for each package;

2> Save to local

Copy the updated fpga.yml from the “west” project to the corresponding local location and save it, with the time stamp used for differentiation.

3. Clear bit file

Change dir to the path that bit file saved, recursively delete the entire directory and all its contents by rmtree.

Note:

1> There will be an error without this step when calling IntegrationTester tool to burn the bit file later.

2> Before performing the deletion, add another level of directory to prevent accidental deletion of other files on the computer’s disk, enhancing security thereby.

4. Download bit file

1> Connect Wiki by account info and page_id, and parse the page HTML and extract the body;

2> Search for the first table, traverse the table rows and extract the attachment;

3> Get the bit file name, and extract the “date” and “version” parameters from it;

4> Compare with the previous Jira about the two parameters: bitfile date and version;

5> If no issue, download bit file to local by calling API download_by_file_name from wiki_helper.py.

5. Generate history.log

Obtain the current time stamp, bit file name, and the absolute path of the previous *.yml file, and write them into the history.log.

6. Call Integration Tools

Call the integration tools recursively for the ap and np dual-core:

1) Call Integration Builder

​ 1> Only Need to call it once for the same core;

​ 2> Do clean before building for another core to avoid interference between generated products;

​ 3> Challenge: How add a new IC to this tool, further to call it to finish building with required

​ in-parameters. Please see the next blogs.

2) Call Integration Tester

​ Challenge: How to integrate a new IC to this tool, and how to add test case into it, further to call it to finish burning and auto testing with required in-parameters. Please see the next blogs.

3) Run test case

​ There are many test cases for each sub_module of each core. Run the test cases one by one and save the test info, which are integrated by Integration Tester tool already, need to pass the XML parameter to inform the tool.

7. Upload test log to FTP

After all tests finished, IO_report and test_log will be generated and saved to designated paths.

Call copy_report API to write report configurations and generate the jira_info_file without any test case, and call copy_log_to_ftp API to achieve uploading function.

8. Report Jira

Call read_test_info and read_report APIs to fill in the jira_info_file again with adding test cases. Then call read_report_with_test_case function to read the test report data automatically, analyze and organize the detailed information and error messages for each test case, generate the input file for the Jira issue description, and support the subsequent automatic Jira creation and annotation process.

Last but not least, call final create_jira_for_autotest function to create Jira:

Connect Jira -> Collect all_success_info and all_fail_info -> Upload *.yml to Jira attachment -> Call create_jira_issue function to create issue and assign Jira tickets to different owners by different sub-modules -> Build total_comment with stat_comment, t_success_comment/universal_comment, “Fail TC(s) xxx” , common_comment, and etc.

Summary

This set of automated testing tools can efficiently assist R&D in conducting automated tests during the FPGA stage and before the new IC is assembled.
